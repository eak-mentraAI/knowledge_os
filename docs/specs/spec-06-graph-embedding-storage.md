# SPEC 06: Graph and Embedding Storage

## Purpose

Persist graph structures and embeddings with provenance, idempotency, and replaceable storage adapters.

## Business-Goal Trace

Supports grounded retrieval, architecture impact analysis, assistant generation, project spaces, and organizational memory preservation.

## Scope

MVP graph target:

- Neo4j.

Supported future graph targets:

- Kuzu.
- Memgraph.
- Neptune.

Embedding storage target will be chosen through a separate decision record after deployment and retrieval constraints are clear.

## Required Behavior

- Write nodes and edges idempotently.
- Preserve source evidence and extraction metadata.
- Export `nodes.json` and `edges.json` artifacts.
- Store embeddings for documents, concepts, components, capabilities, and use cases.
- Keep graph and embedding records linked by stable IDs.

## Interfaces

Input:

- `ResolvedEntity`
- `ResolvedRelationship`
- `EmbeddingRecord`

Output:

- graph mutations
- storage receipts
- export artifacts

## Security and Privacy

- Enforce authorization and corpus boundaries in storage adapters.
- Avoid storing raw secrets in graph properties.
- Support deletion or redaction workflows for sensitive content.

## Performance

- Graph writes must be batched.
- Embedding generation must record cost, latency, and model version.
- Storage adapters must expose health and backpressure signals.

## Acceptance Criteria

- Re-running the same graph write does not duplicate entities or edges.
- Nodes and edges retain evidence references.
- Embeddings can be regenerated by model version without corrupting prior indexes.
